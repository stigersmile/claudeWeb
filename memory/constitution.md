## 项目概述

**项目名称**: claudeWeb

**项目目标**: 构建一个高质量、可维护、可扩展的 Web 应用程序，采用规范驱动开发（Spec-Driven Development）方法论，确保代码质量和开发效率。

**目标用户**: Web 应用程序的最终用户和开发团队

---

## 核心原则

### 原则 1: 规范优先开发

**描述**: 所有功能必须先编写规范文档，明确需求和设计，然后再进行实现。

**规则**:
- 每个新功能必须有完整的规范文档（spec.md）
- 规范必须包含：功能描述、用户故事、技术要求、验收标准
- 未经规范审查的功能不得进入实现阶段
- 规范变更必须通过团队审查

**理由**: 规范优先确保团队对功能有统一理解，减少返工，提高开发效率和代码质量。

### 原则 2: 代码质量标准

**描述**: 所有代码必须符合项目的质量标准，保持高可读性和可维护性。

**规则**:
- 代码必须遵循项目的编码规范和风格指南
- 每个函数和模块必须有清晰的职责和文档注释
- 代码复杂度必须保持在可接受范围内
- 必须进行代码审查（Code Review）才能合并到主分支
- 不允许提交包含 ESLint/TypeScript 错误的代码

**理由**: 高质量的代码减少技术债务，便于团队协作和长期维护。

### 原则 3: 测试驱动

**描述**: 所有功能必须有相应的测试覆盖，确保功能正确性和回归安全。

**规则**:
- 核心业务逻辑必须有单元测试覆盖
- 关键用户流程必须有集成测试或 E2E 测试
- 测试覆盖率目标：核心模块 ≥ 80%
- Bug 修复必须先编写复现测试，再进行修复
- 所有测试必须通过才能合并代码

**理由**: 测试确保代码正确性，防止回归，提高重构信心。

### 原则 4: 安全性第一

**描述**: 安全性是不可妥协的基本要求，必须在设计和实现的每个阶段考虑安全因素。

**规则**:
- 所有用户输入必须进行验证和清理
- 必须防范常见 Web 安全漏洞（XSS、CSRF、SQL 注入等）
- 敏感数据必须加密存储和传输
- 依赖包必须定期更新，及时修复安全漏洞
- 不得在代码中硬编码密钥、密码等敏感信息
- 必须实施适当的身份验证和授权机制

**理由**: 安全漏洞可能导致数据泄露、系统被攻击等严重后果，必须从源头预防。

### 原则 5: 性能优化

**描述**: 应用程序必须提供良好的用户体验，响应时间和资源使用必须在合理范围内。

**规则**:
- 页面首次加载时间目标 < 3 秒
- 交互响应时间目标 < 100 毫秒
- 必须进行代码分割和懒加载优化
- 图片和静态资源必须压缩和优化
- 关键性能指标（Core Web Vitals）必须达标
- 数据库查询必须优化，避免 N+1 问题

**理由**: 性能直接影响用户体验和业务转化率，是产品成功的关键因素。

### 原则 6: 可访问性标准

**描述**: 应用程序必须遵循 Web 可访问性标准，确保所有用户都能使用。

**规则**:
- 必须符合 WCAG 2.1 AA 级标准
- 所有交互元素必须支持键盘操作
- 图片必须有适当的 alt 文本
- 颜色对比度必须符合标准
- 必须支持屏幕阅读器
- 表单必须有清晰的标签和错误提示

**理由**: 可访问性是基本的用户权益，也是产品覆盖更广泛用户群的必要条件。

### 原则 7: 文档完整性

**描述**: 项目必须维护完整、准确、及时的文档，便于团队协作和知识传承。

**规则**:
- 每个功能模块必须有 README 文档
- API 必须有完整的接口文档
- 复杂逻辑必须有设计文档说明
- 文档必须与代码同步更新
- 必须有项目入门指南（Getting Started）
- 必须记录重要的架构决策（ADR）

**理由**: 良好的文档降低学习成本，提高开发效率，确保知识不会因人员变动而丢失。

### 原则 8: 版本控制规范

**描述**: 必须遵循规范的版本控制流程，确保代码历史清晰可追溯。

**规则**:
- 使用 Git 进行版本控制
- 提交信息必须清晰描述变更内容，遵循 Conventional Commits 规范
- 使用功能分支开发，不直接在主分支提交
- 合并前必须进行代码审查
- 主分支必须始终保持可部署状态
- 使用语义化版本号（Semantic Versioning）

**理由**: 规范的版本控制使团队协作更顺畅，问题定位更快速，回滚更安全。

## 治理规则

### 修订程序

1. **提议阶段**: 任何团队成员可以提议宪章修订
2. **讨论阶段**: 团队讨论提议的修订，评估影响
3. **审查阶段**: 项目负责人审查修订提议
4. **批准阶段**: 修订获得批准后更新宪章
5. **生效阶段**: 更新版本号、修订日期，通知所有团队成员

### 版本控制策略

- **MAJOR 版本**: 移除或重新定义核心原则，向后不兼容的重大变更
- **MINOR 版本**: 添加新原则或显著扩展现有原则
- **PATCH 版本**: 澄清说明、措辞优化、错误修正、非实质性改进

### 合规审查

- 每季度进行一次宪章合规性审查
- 在功能规范评审时检查是否符合宪章原则
- 在代码审查时检查是否遵循宪章要求
- 发现违规情况必须及时纠正并记录

### 例外处理

在特殊情况下，可以申请临时例外：
1. 必须有充分的理由和风险评估
2. 必须获得项目负责人批准
3. 必须制定补救计划和时间表
4. 必须在项目文档中记录例外情况
